The distinction

AI should translate rules. The solver should make the schedule.

A language model predicts text. That makes it useful for explanation, code generation, and translating messy human rules into something more precise. It also means it can produce a schedule that looks reasonable while quietly breaking a vacation request, rest rule, or coverage requirement.

A constraint solver works differently. You give it variables, constraints, and an objective. It searches for assignments that satisfy the constraints. If the rules conflict, it can tell you there is no feasible schedule instead of making one up.

The practical pattern is: use AI to help write the constraints, then use CP-SAT to solve the schedule.

This is the part most "just use ChatGPT" posts miss. ChatGPT is not the scheduler. The solver is the scheduler. ChatGPT is the helper you use to write the solver, clean the inputs, explain errors, and add one rule at a time.

The v1 rule set

Start with a small rule set that actually runs

This sample solver is intentionally small. It is not a production anesthesia scheduler. The goal is to prove the workflow before adding real-world complexity.

  1. Cover every required shift in coverage.csv.
  2. Respect shift eligibility in clinicians.csv.
  3. Block hard vacation and no-call requests in requests.csv.
  4. Assign each clinician to at most one shift per day.
  5. Enforce a minimum rest gap between assignments.
  6. Prefer fair total assignment counts.
  7. Prefer fair weekend assignment counts.
  8. Use prior history.csv assignments as part of the fairness picture.
  9. Prefer more rest spacing when the hard rules leave room.
This is enough to get a useful dummy schedule. Real anesthesia groups usually add holiday rotation, partner versus non-partner allocation, site-specific coverage, backup call, post-call recovery, locked assignments, and local exceptions.

Inputs

The four CSV files

The solver reads structured CSVs instead of free-form text. That is what keeps the schedule grounded.

clinicians.csv

Who can be scheduled, which shifts they can cover, target counts, max counts, and rest rules.

coverage.csv

The demand file: each date, each shift type, and how many people are required.

requests.csv

Vacation, no-call requests, and soft preferences. Hard requests block assignments.

history.csv

Prior assignments used to make the new schedule fair against recent history.

File Example row Meaning
clinicians.csv fox,A. Fox,1,1.0,1,1,5,6,2,3,1 A. Fox is active, can cover OR and OB, targets 5 shifts, maxes at 6, targets 2 weekend shifts.
coverage.csv 2026-06-01,OB,1 June 1 needs one OB clinician.
requests.csv r1,cary,2026-06-05,2026-06-07,vacation,1,,Family trip M. Cary cannot be assigned to any shift from June 5 through June 7.
history.csv 2026-05-02,fox,OR,final A. Fox covered OR on May 2, which counts toward fairness history.

Demand generation

coverage.csv is the list of shifts the solver must fill

Yes: coverage.csv is basically the thing you need to fill out. It is the demand side of the schedule. If a row exists, the solver tries to staff it. If a row is missing, the solver does not know that shift exists.

You do not have to type it by hand. In fact, you probably should not. If your coverage pattern is regular, have a small script generate it.

Natural language to coverage.csv

You can tell an agent:

Create a Python script that writes coverage.csv for June 2026.
Rules:
- Every Saturday and Sunday needs one NORTH call and one SOUTH call.
- Weekdays do not need call coverage in this example.
- Output columns must be: date,shift_type,required_count.
- Dates must be ISO format YYYY-MM-DD.

That script should produce rows like:

date,shift_type,required_count
2026-06-06,NORTH,1
2026-06-06,SOUTH,1
2026-06-07,NORTH,1
2026-06-07,SOUTH,1

The same pattern works for more complex demand:

  • one OB call every day
  • two OR calls on weekdays and one OR call on weekends
  • different sites like NORTH, SOUTH, and OB
  • holiday-specific call requirements
  • higher required counts on known heavy days
When you add a new shift type to coverage.csv, add a matching eligibility column in clinicians.csv. For example, shift type NORTH expects can_north.

Customization

After the sample works, copy the data and change one thing at a time

Make your own data folder:

cp -R data/sample data/my_data

Then create your own config file:

cp config/sample_rules.json config/my_rules.json

In config/my_rules.json, change:

{
  "input_dir": "data/my_data",
  "output_csv": "output/my_schedule.csv"
}

Run it:

.venv/bin/python solver.py --config config/my_rules.json

Change one file at a time. First clinicians. Then coverage. Then requests. Then history. If the model becomes infeasible, the newest change is usually where to start looking.

Privacy

Do not paste private schedules into public tools

Even if there is no patient information, physician schedules, vacation requests, and internal group rules can still be sensitive. Start with dummy data. Replace names with IDs before testing your own setup.

  • Do not upload real call history to public GitHub issues.
  • Do not paste real vacation requests into tools unless your group allows it.
  • Check retention and training settings before using an AI product.
  • Use synthetic examples when asking for help.

See docs/agent-privacy.md for a longer checklist.

Where it breaks

This gets hard when the rules become real

The basic pattern is not magic. The hard part is making it work for an actual group: clean requests, historical fairness import, local exceptions, infeasible schedules, post-call recovery, locked assignments, holiday rotation, secure access, readable schedule views, distribution, and month-by-month maintenance.

That is where a maintained product starts to make sense. A real schedule needs more than a local script: requests in one place, history tracked over time, secure access, review before publishing, and a cleaner way to distribute the result to the group.
